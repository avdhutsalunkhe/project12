# Conversational SHL Assessment Recommender
## Technical Approach & Evaluation Report

### 1. System Architecture & Design Choices
The goal of this project was to build a robust, conversational API that accurately recommends SHL assessments based on natural-language inputs.

**Stateless Architecture:**
We chose a purely stateless REST architecture (`POST /chat`) using FastAPI. Rather than relying on a heavy backend database (like Redis or PostgreSQL) to track user session states, the API requires the client to send the entire `messages` history payload on each request. The `ConversationState` engine deterministically reconstructs the user’s intent (role, skills, seniority, duration constraints) by re-processing the history. This ensures that the backend can instantly scale horizontally on Render without session-stickiness issues.

**Zero Hallucination Guarantee:**
To guarantee that the AI never invents non-existent SHL assessments, we completely removed generative LLMs (like GPT-4) from the final recommendation generation step. Instead, we load the canonical `catalog.json` into memory on startup via the `HallucinationGuard`. Any recommendation generated by the system is strictly validated against this in-memory set. If a URL or Name does not perfectly match the official catalog, it is blocked.

### 2. Retrieval Setup (RAG)
We implemented a **Hybrid Retrieval (RAG) Pipeline** utilizing local vector embeddings and keyword matching, ensuring maximum data privacy and low latency without incurring API costs.

1. **Semantic Search:** We embedded the entire SHL catalog using the open-source `sentence-transformers/all-MiniLM-L6-v2` model and stored it locally in DuckDB via ChromaDB. This captures the deep semantic meaning of queries (e.g., matching "coding" to "programming").
2. **Keyword & Metadata Search:** Semantic search struggles with highly specific technical terms (e.g., "C++" vs "C#"). We implemented a TF-IDF keyword overlap system and strict metadata filters (e.g., Maximum Duration, Seniority Match) to filter the vector store results.
3. **Hybrid Ranker:** The results from both systems are merged using a mathematically weighted formula: `(0.45 * Semantic) + (0.25 * Keyword) + (0.20 * Role) + (0.10 * Seniority)`.

### 3. Prompt Design & Conversational Flow
Because we utilize a deterministic extraction pipeline, we implemented a **Dynamic Clarification Engine** rather than a conversational LLM prompt.

When a query is received, the `ConversationState` calculates a `sufficiency_score` based on how many dimensions were successfully extracted (Role, Seniority, Skills, etc.). If the score falls below a threshold (0.55), the engine intercepts the retrieval phase and automatically generates a targeted clarification question (e.g., *"I see you need a Python test, but what seniority level are you hiring for?"*). This guarantees high-accuracy results by refusing to guess vague queries.

### 4. What Didn't Work
Initially, we attempted to use a generative LLM to format the final output and select the assessments. However, we discovered two major flaws:
1. **Hallucinations:** The LLM would occasionally invent URLs (e.g., `shl.com/python-advanced`) that looked legitimate but resulted in 404 errors.
2. **Latency:** Calling an external API added 2-3 seconds of latency per turn.

By migrating the extraction to regex/pattern-matching and the retrieval to a local ChromaDB Hybrid Ranker, we dropped the response time from 3000ms to ~150ms while completely eliminating hallucinations.

### 5. Evaluation Method & Measuring Improvement
To systematically measure improvement, we developed an **Offline Evaluation Framework** (`tests/evaluate_recall.py`) running against a golden dataset (`golden_queries.json`).

We specifically measured two metrics:
*   **Recall@10:** Measures if the mathematically optimal assessment appears in the top 10 results returned by the Hybrid Ranker.
*   **Groundedness:** Measures the percentage of outputted assessments that actually exist in the SHL catalog.

By tuning the weights in our Hybrid Ranker (shifting from 0.80 Semantic to 0.45 Semantic / 0.25 Keyword), our Recall@10 improved significantly for technical skill queries, while Groundedness remained mathematically locked at 100% due to the `HallucinationGuard`.
